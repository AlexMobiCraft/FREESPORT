## User Story

Как 1С Manager,
я хочу, чтобы сумма документа в XML-экспорте заказа совпадала с суммой строк `<Товары>`,
чтобы интеграция с 1С не падала на валидации, когда у заказа есть стоимость доставки.

---

## Контекст и суть дефекта (tech-debt #16)

В `OrderExportService._create_document_element` сумма документа берётся из `order.total_amount`:

```python
# backend/apps/orders/services/order_export.py:172
self._add_text_element(document, "Сумма", self._format_price(order.total_amount))
```

А блок `<Товары>` (`_create_products_element`, строка 325) содержит только физические товары — сумма строк `<Товар>/<Сумма>` равна сумме `item.total_price`. При наличии доставки документ `<Сумма>` ≠ сумма строк → 1С падает на валидации.

### Уточнение по актуальной модели данных (КРИТИЧНО — AC эпика устарел)

Анализ кода выявил, что **AC эпика написан под устаревшую модель** (до разделения на мастер/субзаказы, Story 34-x):

- `OrderExportService` экспортирует **только субзаказы** (`is_master=False`) — `views.py:585-588`. Мастер-заказы в 1С не выгружаются.
- При создании заказа (`order_create.py`) `delivery_cost` пишется **только в мастер-заказ** (строка 95), а всем субзаказам присваивается `delivery_cost = Decimal("0")` (строка 119).

**Следствие:** в текущем коде стоимость доставки **вообще не попадает в 1С** — у экспортируемых субзаказов она всегда `0`. Буквальная проверка `order.delivery_cost` на субзаказе была бы мёртвым кодом.

### Принятое решение (согласовано с заказчиком 2026-05-18)

Доставка облагается **НДС 22%** и должна попадать в субзаказ, относящийся к организации/складу со ставкой 22%. Реализация:

1. **`delivery_cost` маршрутизируется в реальный субзаказ с `vat_group=22`** при создании заказа (а не остаётся только в мастере).
2. Если среди VAT-групп корзины **нет группы 22%** — создаётся **дополнительный реальный субзаказ** `vat_group=22` с единственным наполнением — доставкой (без `OrderItem`).
3. Экспорт читает `order.delivery_cost` напрямую (поле модели **уже существует** — миграция не нужна) и добавляет в `<Товары>` виртуальную строку «Доставка».

Это делает буквальный AC эпика снова корректным — экспорт проверяет `order.delivery_cost`, и оно теперь ненулевое на нужном субзаказе.

> **Почему реальный субзаказ, а не виртуальный документ при экспорте:** реальная строка `Order` единообразно участвует в отслеживании `sent_to_1c`, нумерации субзаказов и логике «пометить мастер, когда все `sub_orders` отправлены» (`views.py:477-494`). Синтетический документ потребовал бы отдельного механизма для всего этого.

## Acceptance Criteria

### AC-1: Доставка маршрутизируется в субзаказ 22% при создании заказа

**Given** заказ оформляется с `delivery_cost > 0`,
**When** `OrderCreateService` создаёт мастер и субзаказы,
**Then** `delivery_cost` записывается в субзаказ с `vat_group == 22`:
- если такой субзаказ есть — в первый из них (по `suborder_sequence`), его `total_amount` увеличивается на `delivery_cost`;
- если субзаказа с 22% нет — создаётся дополнительный субзаказ `vat_group=22`, `delivery_cost=X`, `total_amount=X`, **без `OrderItem`**.

**And** инвариант сохраняется: `master.total_amount == sum(sub.total_amount) − discount_amount`.

### AC-2: Виртуальная позиция «Доставка» в XML

**Given** субзаказ с `delivery_cost > 0`,
**When** `OrderExportService` формирует XML,
**Then** в блок `<Товары>` добавляется виртуальная позиция «Доставка»: `Количество=1`, `ЦенаЗаЕдиницу` и `Сумма` равны `delivery_cost`, НДС 22% «в том числе» (`УчтеноВСумме=true`).

### AC-3: Сумма документа сходится с суммой строк

**Given** экспортированный XML субзаказа с доставкой,
**When** проверяется сумма документа `<Документ>/<Сумма>`,
**Then** она равна сумме всех строк `<Товары>/<Товар>/<Сумма>`, включая позицию «Доставка».
**And** `<Документ>/<Налоги>/<Налог>/<Сумма>` (НДС документа) включает НДС доставки — валидация 1С проходит.

### AC-4: Заказ без доставки не меняется

**Given** заказ с `delivery_cost = 0` (например, самовывоз),
**When** создаётся заказ и формируется XML,
**Then** дополнительный субзаказ не создаётся, виртуальная позиция «Доставка» не добавляется — поведение идентично текущему.

### AC-5: Полный цикл экспорта покрыт тестами

**Given** интеграционный тест экспорта заказа,
**When** он выполняется,
**Then** покрывает оба случая — заказ с доставкой (есть 22%-группа / нет 22%-группы) и без доставки.

## Анализ кодовой базы (что и где менять)

### `backend/apps/orders/services/order_create.py`

Текущий поток (строки 85-153): мастер получает `delivery_cost`, каждый субзаказ — `delivery_cost=Decimal("0")`, `total_amount=group_total`.

**Изменить:**
- После сортировки `ordered_groups` определить, есть ли группа со ставкой `Decimal("22")` (ключ группы — `(vat_key, warehouse_key)`, `vat_key` — это `Decimal`).
- В цикле создания субзаказов: первому субзаказу с `vat_key == Decimal("22")` присвоить `delivery_cost=delivery_cost` и `total_amount = group_total + delivery_cost`. Остальным — `0`/`group_total` как сейчас.
- Если ни одной 22%-группы нет и `delivery_cost > 0` — после цикла создать дополнительный субзаказ: `vat_group=Decimal("22")`, `delivery_cost=delivery_cost`, `total_amount=delivery_cost`, `suborder_sequence = len(ordered_groups) + 1` (через `OrderNumberingService.build_suborder_number`), `OrderItem` для него не создаются.
- При `delivery_cost == 0` — поведение строго как сейчас (доп. субзаказ не создаётся).
- Мастер по-прежнему хранит `delivery_cost` для клиентского итога.

> `build_suborder_number(master, sequence)` принимает любой `sequence >= 1` — номер доп. субзаказа формируется штатно (`order_numbering.py:138-154`).

### `backend/apps/orders/services/order_export.py`

**`_validate_order` (строки 148-155)** — сейчас пустой заказ (`not items_queryset`) пропускается. Изменить: субзаказ без товаров, но с `delivery_cost > 0`, считать валидным.

```python
def _validate_order(self, order: "Order") -> bool:
    items_queryset = order.items.all()
    if not items_queryset and order.delivery_cost <= 0:
        logger.warning(f"Order {order.order_number}: no items and no delivery, skipping")
        return False
    return True
```

**`_create_products_element` (строки 325-401)** — после цикла по `order.items.all()` добавить виртуальную строку «Доставка», если `order.delivery_cost > 0`:
- `<Наименование>` — «Доставка».
- `<Ид>` — стабильный идентификатор номенклатуры 1С (см. «Открытый вопрос» ниже).
- `<ЦенаЗаЕдиницу>` и `<Сумма>` = `_format_price(order.delivery_cost)`, `<Количество>` = `1`.
- `<Налоги>/<Налог>`: «НДС», `УчтеноВСумме=true`, `Ставка=22`, `Сумма` = `_calc_vat_amount(order.delivery_cost, Decimal("22"))`.
- `<ЗначенияРеквизитов>`: `ВидНоменклатуры` / `ТипНоменклатуры` = «Услуга» (доставка — услуга, не товар).
- `<Действие>` (резервирование) для услуги доставки **не добавлять**.

**`_get_document_vat_amount` (строки 416-424)** — сейчас суммирует НДС только по `order.items`. Добавить слагаемое НДС доставки при `order.delivery_cost > 0`: `+ _calc_vat_amount(order.delivery_cost, Decimal("22"))`. Иначе `<Документ>/<Налоги>` разойдётся со строками (нарушит AC-3).

**Документ `<Сумма>` (строка 172)** — менять **не нужно**: `order.total_amount` экспортируемого субзаказа уже включает доставку (после правки `order_create.py`).

**`_get_order_vat_rate`** — для доп. субзаказа `vat_group=22` задан, метод вернёт `Decimal("22")` сразу (строка 436-437). Для существующего 22%-субзаказа — тоже 22. Изменений не требует.

### Не менять

- `MEDIA_ROOT`, импорт 1С — это Story 36.1.
- Логику организации/склада (`_get_org_and_warehouse`, `WAREHOUSE_RULES`) — субзаказ `vat_group=22` штатно отмаппится через `ORGANIZATION_BY_VAT[22]`.
- Модель `Order` — поле `delivery_cost` уже есть (`models.py:170-179`), **миграция не нужна**.

## Открытый вопрос для разработчика (требует входных данных по 1С)

`<Ид>` виртуальной позиции «Доставка». Если в 1С УТ 11 уже заведена номенклатурная позиция «Доставка» с фиксированным GUID — её нужно подставлять, иначе 1С при каждом импорте может создавать новую номенклатуру.

**Рекомендация:** вынести идентификатор в настройку `ONEC_EXCHANGE["DELIVERY_NOMENCLATURE_ID"]` (`base.py`). До получения реального GUID от стороны 1С использовать стабильный синтетический fallback (`"delivery-service"`). Уточнить GUID у ответственного за 1С перед выкаткой на production.

Аналогично — ставку НДС доставки вынести как `ONEC_EXCHANGE["DELIVERY_VAT_RATE"]` с `default = Decimal("22")` (по согласованному решению — 22%).

## Тесты

### Unit — `OrderExportService` (`backend/tests/unit/test_order_export_service.py`)

- Субзаказ с `delivery_cost > 0`, есть товары: в `<Товары>` есть строка «Доставка»; сумма строк == `<Документ>/<Сумма>`.
- Субзаказ-доставка без `OrderItem` (`delivery_cost > 0`): не пропускается `_validate_order`; XML содержит только строку «Доставка»; документ валиден.
- Субзаказ с `delivery_cost = 0`: строки «Доставка» нет (регресс AC-4).
- `<Документ>/<Налоги>/<Налог>/<Сумма>` == сумма НДС всех строк, включая доставку.

### Unit — `OrderCreateService`

- Корзина с группой 22%: `delivery_cost` попал в первый 22%-субзаказ, его `total_amount` увеличен; доп. субзаказ не создан.
- Корзина без группы 22% (например, только 5%): создан доп. субзаказ `vat_group=22`, `delivery_cost=X`, без `OrderItem`.
- `delivery_cost = 0`: доп. субзаказ не создаётся; число субзаказов == числу VAT-групп.
- Инвариант `master.total_amount == sum(sub.total_amount) − discount`.

### Integration / E2E

**Каталог:** `backend/tests/integration/` (рядом с `test_onec_export.py`, `test_onec_export_e2e.py`).

- Полный цикл: создать заказ с доставкой → выгрузить XML через `OrderExportService` → распарсить → проверить, что для каждого `<Документ>` сумма строк == `<Сумма>` и НДС документа сходится.
- Покрыть три сценария: доставка + 22%-группа есть; доставка + 22%-группы нет; без доставки.

> Тесты экспорта используют фабрики заказов/товаров (синтетические товары здесь допустимы — запрет синтетики касается **XML импорта 1С**, не unit-фабрик заказов). Изоляция БД — штатная (`TRUNCATE CASCADE` autouse).

## Примечания для разработчика

1. **AC эпика в `epics.md` неточен** — он сформулирован до Story 34-x. Реализовывать по AC **этой story** (раздел выше), а не по дословному тексту эпика.
2. **Самая частая ошибка — забыть про `_get_document_vat_amount`.** Если добавить строку «Доставка» в `<Товары>`, но не учесть её НДС в `<Документ>/<Налоги>` — 1С всё равно упадёт на сверке НДС. Правь оба метода.
3. **Миграция не нужна** — `delivery_cost` уже есть у `Order`. Меняется только логика заполнения.
4. **`vat_group` сравнивать как `Decimal`** — в `order_create.py` ставки это `Decimal`. Сравнивай с `Decimal("22")`, не с `int` 22 и не строкой.
5. **Несколько 22%-субзаказов** (разные склады, одна ставка) — доставку класть в первый по `suborder_sequence` (согласовано с заказчиком).
6. **Доп. субзаказ-доставка без `OrderItem`** — обязательно проверь, что `_validate_order` его пропускает, иначе он вылетит из экспорта и доставка опять не дойдёт до 1С.
7. **`<Ид>` доставки** — см. «Открытый вопрос». Без реального GUID 1С может плодить дубли номенклатуры; синтетический fallback допустим только до выяснения.
8. Прогони полный e2e экспорта и убедись в сходимости сумм — это единственная надёжная проверка против регрессии валидации 1С.
