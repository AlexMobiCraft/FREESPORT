# Техническое задание для AI-агента

## Тема

**Разработка технической спецификации новой схемы нумерации заказов FREESPORT**

## Цель задачи

Подготовить полноценную техническую спецификацию для внедрения новой схемы нумерации заказов, включая:

- генерацию номеров мастер-заказов
- генерацию номеров субзаказов
- правила канонического хранения
- правила UI-отображения
- нормализацию поиска в админке
- требования к модели данных, API, backend-логике и frontend-formatting
- оценку рисков, миграций и тестового покрытия

## Важно

На данном этапе агенту **не нужно писать production-код**.
Задача агента — **разработать и оформить спецификацию**, достаточную для последующей реализации другим разработчиком или агентом.

## Контекст

В текущем проекте нумерация заказов должна быть упрощена и стандартизирована.
Новая схема должна заменить текущий подход и стать единым источником истины для:

- backend
- frontend
- административного интерфейса
- email/документов
- внешних интеграций, включая `1С`

## Зафиксированные решения, которые нельзя пересматривать

Агент должен использовать следующие правила как **обязательные и уже утверждённые**.

### 1. Мастер-заказ

- **Канонический номер заказа**
  - `CCCCCYYNNN`

- **Каноническое хранение**
  - только полный вид, например:
  - `0462026007`

- **UI-отображение**
  - `CCCC-YYNNN`
  - пример:
  - `4620-26007`

- **Правило UI для `CCCC`**
  - это **последние 4 цифры** блока `CCCCC`

- **`CCCCC`**
  - отдельный неизменяемый `customer_code`
  - длина `5`
  - только цифры

- **`NNN`**
  - порядковый номер заказа клиента в рамках календарного года
  - диапазон `001..999`

- **Генерация**
  - только атомарно в транзакции

- **Изменяемость**
  - номер после создания не меняется

### 2. Гости

- **Гостевые заказы запрещены**
- агент должен учитывать, что заказ может создать только авторизованный пользователь с `customer_code`

### 3. Субзаказы

- **Канонический номер субзаказа**
  - `CCCCCYYNNNS`

- **UI-отображение субзаказа**
  - `CCCCC-YYNNN-S`

- `S` — порядковый номер субзаказа в рамках мастер-заказа

Примеры:

- мастер:
  - `0462026007`
- субзаказ 1:
  - `04620260071`
- UI субзаказа:
  - `04620-26007-1`

### 4. Поиск в админке

Поиск должен принимать:

- **канонический формат**
  - `0462026007`

- **UI-формат мастер-заказа**
  - `4620-26007`

И автоматически нормализовать ввод к каноническому виду:

- `0462026007`

Для субзаказов агент также должен описать нормализацию:

- `04620-26007-1` -> `04620260071`

## Основная задача агента

Подготовить спецификацию, которая ответит на следующие вопросы:

- как именно хранится `customer_code`
- где и как генерируется `order_number`
- как обеспечивается атомарность генерации
- как различаются мастер-заказы и субзаказы
- как работает форматирование номера для UI
- как работает нормализация номера для поиска в админке
- какие данные и ограничения должны быть отражены в моделях
- какие изменения потребуются в backend, admin, API, frontend и тестах
- как внедрить новую схему безопасно для существующих заказов и интеграций

## Что агент должен проработать в спецификации

### 1. Data model

Нужно описать:

- требуется ли отдельное поле `customer_code` в модели пользователя
- требования к `customer_code`
- как и когда он назначается
- можно ли его менять
- какие уникальные ограничения нужны
- нужны ли отдельные служебные поля в заказе:
  - `customer_code_snapshot`
  - `order_year`
  - `customer_year_sequence`
  - `suborder_sequence`

### 2. Генерация номеров

Нужно описать:

- точный алгоритм генерации мастер-заказа
- точный алгоритм генерации субзаказа
- в каком слое это должно жить:
  - модель
  - сервис
  - domain helper
  - отдельный generator class
- как избежать коллизий при параллельном создании заказов
- нужен ли отдельный счётчик по паре:
  - `customer_code`
  - `year`

### 3. Админка

Нужно описать:

- как админка принимает `4620-26007`
- как происходит нормализация в `0462026007`
- как поддержать поиск по субзаказу
- как сделать поведение предсказуемым для менеджеров
- какие формы ввода допустимы
- какие формы ввода считаются невалидными

### 4. UI / frontend

Нужно описать:

- единый formatter для мастер-заказов
- единый formatter для субзаказов
- где он должен использоваться
- должен ли frontend когда-либо отправлять UI-номер обратно на backend
- как отображать номер в:
  - списке заказов
  - карточке заказа
  - деталях заказа
  - PDF / email / уведомлениях, если применимо

### 5. Интеграции

Нужно проверить и описать влияние новой схемы на:

- `1С`
- email-шаблоны
- экспорт/импорт заказов
- логи
- admin actions
- API-контракты
- возможные внешние consumer’ы `order_number`

### 6. Миграция и backward compatibility

Нужно определить:

- меняются ли номера у старых заказов
- как coexist будут жить старые и новые номера
- нужен ли feature flag
- нужен ли rollout поэтапно
- как обрабатывать существующие UUID-like или legacy order numbers
- как не сломать поиск, экспорт и историю заказов

## Ограничения и обязательные допущения

- **Не менять утверждённый формат**
- **Не предлагать гостевые заказы**
- **Не заменять `customer_code` на `user.id` без отдельного обоснования**
- **Не предлагать генерацию через `count() + 1` без защиты от гонок**
- учитывать лимит:
  - `NNN` = максимум `999` заказов клиента в год
- отдельно описать, что происходит при переполнении `NNN`

## Что должно быть на выходе

Агент должен подготовить следующие артефакты.

### 1. Основная спецификация

Документ с разделами:

- цель
- терминология
- формат номера
- правила генерации
- правила отображения
- правила поиска и нормализации
- модель данных
- влияние на backend
- влияние на frontend
- влияние на admin
- влияние на интеграции
- миграционная стратегия
- риски
- тестовая стратегия

### 2. Impact analysis

Краткая таблица:

- компонент
- что меняется
- критичность
- тип изменения

Компоненты минимум:

- `Order model`
- `User model`
- order create flow
- sub-order flow
- Django admin
- serializers / API
- frontend UI
- email/templates
- 1C integration
- tests

### 3. Acceptance criteria

Отдельный блок с проверяемыми критериями готовности реализации.

### 4. Open questions

Если останутся неоднозначности в кодовой базе, агент должен оформить их отдельным списком, а не принимать молчаливые решения.

## Критерии готовности спеки

Спецификация считается готовой, если:

- **[структура]** содержит все ключевые разделы
- **[точность]** не противоречит утверждённым бизнес-правилам
- **[реализуемость]** по ней можно начинать implementation task без доуточнения базовой логики
- **[границы]** чётко отделены master orders, sub-orders, admin search и UI-formatting
- **[совместимость]** описано влияние на legacy-заказы и интеграции
- **[надёжность]** описана атомарная защита от коллизий
- **[тесты]** описаны необходимые unit/integration/regression сценарии

## Что не входит в задачу

- написание production-кода
- создание миграций
- изменение существующих моделей и сериализаторов
- правка frontend-компонентов
- запуск тестов
- обновление админки в коде

Если агент считает, что без кода невозможно уточнить часть поведения, он должен:

- исследовать существующую кодовую базу
- зафиксировать выводы
- оформить рекомендации в спецификации

## Формат работы агента

- сначала исследовать текущую реализацию заказов, субзаказов, admin search и интеграции с `1С`
- затем подготовить спецификацию
- в спорных местах не додумывать молча, а явно фиксировать:
  - допущение
  - риск
  - рекомендацию

## Краткая формулировка задачи для передачи агенту

Разработай техническую спецификацию новой схемы нумерации заказов FREESPORT на основе уже утверждённых правил: мастер-заказ `CCCCCYYNNN`, UI master `CCCC-YYNNN`, субзаказ `CCCCCYYNNNS`, UI suborder `CCCCC-YYNNN-S`, отдельный неизменяемый `customer_code`, атомарная генерация в транзакции, запрет гостевых заказов, поддержка нормализации UI-форматов в админке. Не реализуй код. Подготовь implementation-ready spec, impact analysis, acceptance criteria, migration considerations и список открытых вопросов.
