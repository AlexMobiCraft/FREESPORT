# ADR-010: Публичная проекция дерева категорий 1С

## Статус

Принято

## Дата

2026-05-05

## Контекст

После полной загрузки `groups.xml` и `goods.xml` из 1С витрина могла получить некорректное дерево категорий:

- якорная категория `СПОРТ` не сохранялась как корень в БД;
- дети `СПОРТ` становились `parent=None`, поэтому исходная структура хранения смешивалась с публичной навигацией;
- неизвестные `category_id` из `goods.xml` могли создавать публичные категории-заглушки вида `Категория <uuid>`;
- главная страница и `/catalog` зависели от эвристики фронтенд-потребителя, а не от явного backend-контракта.

Старый подход восстановления `cleanup_root_categories` соответствовал прежнему допущению: удалить `СПОРТ`, поднять его детей в корень и убрать посторонние корни. После анализа реального импорта это допущение признано неверным: FREESPORT должен хранить полное дерево 1С, а витрина должна получать отдельную публичную проекцию.

## Решение

Мы разделяем модель хранения и модель публичной навигации:

1. `Category` хранит полную исходную иерархию из 1С, включая корневой якорь `СПОРТ` (`parent=None`) и его `onec_id`.
2. `/api/v1/categories-tree/` является публичной API-проекцией каталога: эндпоинт возвращает прямых детей активного `СПОРТ` как корневые узлы витрины.
3. Эндпоинт `/api/v1/categories-tree/` не пагинируется и рекурсивно исключает технические категории:
   - `uncategorized`;
   - `onec-unresolved-category`;
   - `Без категории`;
   - ветки-заглушки вида `Категория <uuid>` и старые форматы `Категория <id>`.
4. Фронтенд-потребители (`QuickLinksSection`, `/catalog`) используют `/categories-tree/` как уже готовое публичное дерево и не применяют эвристику "один root => показать children".
5. Неизвестный `category_id` из `goods.xml` не создаёт публичную категорию. Такие товары изолируются в скрытой fallback-категории `onec-unresolved-category`, а счётчик фиксируется в `category_fallbacks`.
6. Для восстановления уже испорченной БД используется `fix_category_tree_public_roots`, а не `cleanup_root_categories`.

## Последствия

### Положительные

- Повторная полная заливка из 1С сохраняет исходное дерево и не превращает дочерние категории `СПОРТ` в технические корневые узлы сайта.
- Витрина получает стабильный контракт: публичные корневые узлы всегда приходят из `/api/v1/categories-tree/`.
- Категории-заглушки и fallback-узлы не протекают в меню главной страницы и `/catalog`.
- Команда восстановления может исправлять старые некорректные данные без каскадного удаления полезных товаров.

### Ограничения

- Код, которому нужна исходная иерархия `Category`, должен явно использовать `/categories/` или прямой доступ к модели, а не `/categories-tree/`.
- Операторы не должны запускать `cleanup_root_categories` как восстановление для нового контракта: она удаляет обязательный якорь `СПОРТ`.
- Если в БД обнаружено несколько активных корневых якорей `СПОРТ`, публичное API отдаёт объединение их детей как временную защиту, но данные нужно выровнять процедурой восстановления.
