### Необходимые изменения:

1.  **[Критично] Уточнить логику команды (AC4):** Текущая команда `import_catalog_from_1c --file-type=goods` выглядит так, как будто она обрабатывает и XML-файлы, что противоречит цели импорта "только изображений". Пожалуйста, модифицируйте команду (например, добавьте флаг `--images-only`), чтобы гарантировать, что она обрабатывает **только** изображения и не затрагивает данные о товарах.
2.  **[Риск] Рефакторинг задачи Celery:** Функция `run_selective_import_task` использует `return` внутри цикла. Это потенциальный баг, который не позволит выполнить несколько типов импорта одновременно. Логику необходимо переработать для корректной обработки всех выбранных типов за один запуск.
3.  **[Улучшение] Детализировать отчетность (AC5):** Статистику следует сделать более подробной. Пожалуйста, расширьте `report_details`, включив списки имен файлов для пропущенных изображений (например, по причине дублирования или отсутствия товара), чтобы упростить диагностику.
4.  **[Вопрос] Политика обновления главного изображения (AC7):** История утверждает, что `main_image` никогда не обновляется. Пожалуйста, рассмотрите, нужна ли нам опция (например, чекбокс), чтобы позволить администраторам принудительно перезаписывать главное изображение при импорте.

После внесения этих исправлений история будет готова к утверждению.

## User Story

**Как** администратор системы
**Я хочу** импортировать только изображения товаров без повторного импорта всего каталога
**Чтобы** быстро обновлять изображения существующих товаров при получении новых фото от поставщиков

---

## Business Context

### Проблема

В настоящее время импорт изображений товаров возможен только при полном импорте каталога (Story 9.6, тип "Полный каталог"). Это приводит к:

- Излишней нагрузке на систему при обработке всего каталога
- Длительному времени выполнения (когда нужно обновить только фото)
- Риску перезаписи данных каталога при необходимости обновить только изображения

### Решение

Добавить отдельный тип импорта "Изображения" на странице импорта данных из 1С, который:

- Обрабатывает только файлы изображений из `goods/import_files/`
- Обновляет только поля `main_image` и `gallery_images` у существующих товаров
- Использует существующую логику импорта изображений из Story 3.1.2
- Валидирует наличие товаров в БД перед импортом

### Бизнес-ценность

- ⚡ Ускорение процесса обновления изображений (минуты вместо часов)
- 🎯 Точечное обновление без риска изменения данных каталога
- 📊 Прозрачная статистика по импорту изображений
- 🔄 Удобство для операторов при работе с фото от поставщиков

- `celery_task_id` установлен

**And** команда обрабатывает только `goods/goods_*.xml` и изображения из `goods/import_files/`

**Given** импорт завершен со статистикой изображений
**When** администратор открывает детали ImportSession

- "Скопировано изображений: 120"

**Given** у товара уже установлены изображения
**When** запускается импорт изображений повторно
**Then** применяется логика из Story 3.1.2:

- `main_image` **НЕ меняется** (остается первое установленное)

- Товары без `onec_id` в XML **пропускаются** (не создаются placeholder)

## Implementation Notes

### Критические моменты

1. **Поведение при повторном импорте**
   - `main_image` **НЕ меняется** если уже установлен (логика из Story 3.1.2)
   - Новые изображения **добавляются** в `gallery_images`
   - Дубликаты **пропускаются** (проверка по имени файла)

2. **Обработка отсутствующих товаров**
   - Если в `goods.xml` есть товар с `onec_id` которого нет в БД → **пропускается**
   - Placeholder товары **НЕ создаются** при импорте только изображений
   - В логах появляется предупреждение: `"Товар с onec_id={uuid} не найден, изображения пропущены"`

3. **Валидация наличия товаров**
   - Проверка `Product.objects.exists()` выполняется **ДО** запуска Celery задачи
   - Предотвращает создание ImportSession если каталог пуст
   - Улучшает UX - пользователь сразу видит проблему

4. **Redis Lock**
   - Используется существующий механизм из Story 9.6
   - Ключ: `import_1c:lock`
   - TTL: 3600 секунд (1 час)
   - Предотвращает параллельные импорты любых типов

5. **Статистика изображений**
   - Сохраняется в `ImportSession.report_details["stats"]`
   - Поля: `images_copied`, `images_skipped`, `images_errors`
   - Отображается в Django Admin detail view (AC6)

### Рекомендации по разработке

1. **Начать с миграции** - создать `ImportType.IMAGES` первым делом
2. **Использовать существующую логику** - не дублировать код из Story 3.1.2
3. **Тестировать с реальными данными** - использовать файлы из `data/import_1c/`
4. **Проверить все AC** - каждый критерий должен иметь автоматический тест
5. **Документировать изменения** - обновить README если нужно

2. **Файлы изображений отсутствуют**

## Questions & Answers

**Q1: Почему не использовать отдельную команду для импорта изображений?**
**A1**: Логика импорта изображений уже интегрирована в `import_catalog_from_1c` (Story 3.1.2). Использование `--file-type=goods` позволяет импортировать только метаданные товаров и изображения, без создания новых товаров.

**Q2: Что происходит если товар удален из БД, но его изображения остались на диске?**
**A2**: При импорте изображений товар с `onec_id` не найден → изображения пропускаются. Старые файлы на диске НЕ удаляются (требуется отдельная задача очистки).

**Q3: Можно ли импортировать изображения для конкретного товара?**
**A3**: В текущей реализации - нет. Импортируются все изображения из `goods/import_files/`. Фильтрация по конкретному товару - future enhancement.

**Q4: Как обрабатываются изображения в разных форматах (JPG, PNG, WEBP)?**
**A4**: Логика из Story 3.1.2 копирует любые файлы из `import_files/` без проверки формата. Валидация формата происходит на уровне Django ImageField при сохранении модели.
