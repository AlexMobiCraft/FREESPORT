# Story 11.0: Маркетинговые флаги для Product model (Подготовка для бейджей)

## Status

**Done**

## Story

**As a** backend разработчик,
**I want** добавить маркетинговые флаги (is_hit, is_new, is_sale, is_promo, is_premium, discount_percent) в модель Product,
**so that** frontend может отображать соответствующие бейджи на карточках товаров и фильтровать товары по этим признакам.

## Business Context

Эта история является **подготовительной** для Story 11.2 "Динамические блоки контента". Без этих полей:

- ❌ Невозможно вручную управлять отображением бейджей "Хит", "Новинка", "Скидка", "Промо", "Премиум"
- ❌ Фильтрация товаров по маркетинговым признакам будет неэффективной
- ❌ Придётся использовать computed logic на frontend (временное решение)

**Зависимые Stories:**

- Story 11.2: Динамические блоки контента (требует эти поля)
- Story 11.3+: Любые фичи с отображением бейджей товаров

## Acceptance Criteria

1. **Добавлены boolean поля в Product model:**
   - `is_hit`: BooleanField для "Хитов продаж"
   - `is_new`: BooleanField для "Новинок"
   - `is_sale`: BooleanField для "Распродажи"
   - `is_promo`: BooleanField для "Акционных товаров"
   - `is_premium`: BooleanField для "Премиум товаров"
   - Все поля имеют `default=False`, `db_index=True`

2. **Добавлено поле для процента скидки:**
   - `discount_percent`: PositiveSmallIntegerField(null=True, blank=True)
   - Валидатор: MaxValueValidator(100)
   - Используется для отображения "% скидки" на бейдже sale

3. **Создана и применена миграция БД:**
   - Миграция добавляет все новые поля
   - Default values заданы для существующих записей
   - Индексы созданы для всех boolean полей (для быстрой фильтрации)

4. **Цвета бейджей синхронизированы с новой сине-голубой палитрой (Story 14.4):**
   - `sale`: bg `#FFE1E8`, text `#E53935`
   - `new`: bg `#E7F3FF`, text `#0060FF`
   - `hit`: bg `#E0F5E8`, text `#00AA5B`
   - `promo`: bg `#FFF0F5`, text `#FF2E93`
   - `discount`: bg `#F0E7FF`, text `#7C3AED`
   - `premium`: bg `#F6F0E4`, text `#6D4C1F`
   - Статусы заказов используют те же токены (deliverd/transit/cancelled)

5. **Обновлены serializers:**
   - ProductListSerializer включает новые поля
   - ProductDetailSerializer включает новые поля
   - Поля доступны только для чтения (read_only=True)

6. **Обновлена OpenAPI спецификация:**
   - api-spec.yaml включает новые поля в Product schema
   - Добавлены примеры для каждого флага
   - Документированы фильтры для новых полей

7. **Добавлены фильтры в ProductViewSet:**
   - Фильтр `is_hit=true` для хитов продаж
   - Фильтр `is_new=true` для новинок
   - Фильтр `is_sale=true` для распродажи
   - Фильтр `is_promo=true` для акций
   - Фильтр `is_premium=true` для премиум товаров
   - Все фильтры работают через ProductFilter (django-filter)

8. **Обновлены тесты:**
   - Unit-тесты для модели Product с новыми полями

## Dev Notes

### Архитектурные решения

**Почему boolean поля вместо JSONB или choices:**

- ✅ **Производительность:** DB indexes для быстрой фильтрации
- ✅ **Гибкость:** Товар может иметь несколько бейджей одновременно
- ✅ **Простота:** Понятная схема для всех разработчиков
- ✅ **Масштабируемость:** Легко добавить новые флаги в будущем

**Альтернативные подходы (не выбраны):**

- ❌ JSONB поле `badges: ["hit", "new"]` - сложнее фильтровать, нет индексов
- ❌ CharField choices - только один бейдж на товар
- ❌ Отдельная таблица ProductBadge - overkill для простых флагов

### Индексы БД

```python
# В Product.Meta.indexes:
models.Index(fields=["is_hit", "is_active"]),        # Для фильтра хитов
models.Index(fields=["is_new", "is_active"]),        # Для фильтра новинок
models.Index(fields=["is_sale", "is_active"]),       # Для фильтра распродажи
models.Index(fields=["is_promo", "is_active"]),      # Для фильтра акций
models.Index(fields=["is_premium", "is_active"]),    # Для фильтра премиум
```

**Обоснование:**

- Composite indexes (is_XXX + is_active) ускоряют типовые запросы
- is_active почти всегда в WHERE clause

### Миграция данных

**Начальное состояние:**

- Все существующие товары: все флаги = False, discount_percent = NULL
- Это безопасно, т.к. default values заданы

**Опциональная настройка после миграции:**

```python

  return (
    <Card>
      {product.is_hit && <Badge variant="hit">Хит продаж</Badge>}
      {product.is_new && <Badge variant="new">Новинка</Badge>}

        </Badge>
      )}
      {product.is_promo && <Badge variant="promo">Акция</Badge>}
      {product.is_premium && <Badge variant="premium">Premium</Badge>}

    </Card>
  );
}

// API запросы с фильтрами:

```

### Приоритет бейджей (если несколько)

Frontend должен показывать **один основной бейдж** по приоритету:

3. **new** - если is_new=true
