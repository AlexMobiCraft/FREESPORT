## Story

**As a** Frontend Developer,
**I want** обновить источники истины для цветовых токенов (design-system.json и globals.css),
**so that** все компоненты могут автоматически использовать новую сине-голубую палитру через CSS-переменные и Tailwind классы.

## Business Value

**Почему это важно:**

1. **Фундамент миграции:** Обновление токенов — первый критический шаг, блокирующий всю остальную работу
2. **Автоматическое применение:** Компоненты использующие токены получат новые цвета без доп. изменений
3. **Единый источник истины:** Централизованное управление цветовой палитрой упрощает поддержку

**Влияние на пользователей:**

- Минимальное (на этом этапе визуальные изменения частичны)
- Полное влияние проявится после Story 14.2

### Tailwind Config Considerations

**Проверка необходимости обновления:**

Story 14.1 обновляет ТОЛЬКО CSS-переменные в `@theme inline` блоке `frontend/src/app/globals.css`.

✅ **Tailwind config НЕ требует изменений** если:

- Проект использует Tailwind CSS 4.0 с `@theme inline`
- Токены определены через CSS-переменные (`var(--color-primary)`)
- Config использует эти переменные без хардкода

⚠️ **Если config содержит хардкод цветов:**

```javascript
// frontend/tailwind.config.ts
theme: {
  extend: {
    colors: {
      primary: '#1F1F1F', // ❌ ХАРДКОД - требует обновления на #0060FF
    }
  }
}
```

**Действие:** Удалить хардкод из config, использовать CSS-переменные.

**Проверка config:**

```bash
# Открыть файл для проверки
cat frontend/tailwind.config.ts | grep -i "primary\|secondary\|#[0-9A-F]"

# Если найдены HEX коды - требуется обновление config
```

## Edge Cases

### Edge Case 1: JSON файл поврежден после редактирования

**Ситуация:** Ошибка синтаксиса в JSON
**Решение:**

1. Откатиться к backup: `cp design-system.json.v2.0.0.backup design-system.json`
2. Использовать JSON linter (VS Code extension)
3. Применить изменения заново

### Edge Case 2: CSS компиляция Tailwind ломается

**Ситуация:** Ошибка после обновления CSS-переменных
**Решение:**

1. Проверить синтаксис: все переменные должны начинаться с `--`
2. Проверить значения: HEX коды должны быть валидными
3. Откатиться к backup globals.css если нужно

### Edge Case 3: Dev server не запускается

**Ситуация:** Ошибка при npm run dev
**Решение:**

1. Проверить Tailwind config: `frontend/tailwind.config.ts`
2. Очистить cache: `rm -rf .next && npm run dev`
3. Проверить node_modules: `npm install`

### Rollback Strategy

Если после обновления обнаружены критические проблемы:

**План А: Rollback через backup (5 минут)**

```bash
# Откат JSON
cp design-system.json.v2.0.0.backup docs/frontend/design-system.json

# Откат CSS
cp globals.css.backup frontend/src/app/globals.css

# Проверка
cd frontend && npm run dev
```

**План Б: Git revert (10 минут)**

```bash
# Найти commit миграции
git log --oneline | grep "feat(design-system)"

# Откатить изменения
git revert <commit-hash>

# Push изменений
git push origin feature/epic-14-color-scheme-migration
```

## QA Results

### QA Review — 2025-11-25 (Quinn)

**Gate Decision:** CONCERNS — критичные требования формулированы не полностью, что создаёт риск неполной миграции и неконсистентного процесса проверки.

#### Findings

1. **Валидационные шаги описаны только для запуска на хостовой ОС.** AC5 и Testing раздел требуют выполнять `node`, `npm run build` и `npm run dev` напрямую на машине разработчика (@docs/stories/epic-14/14.1.design-system-tokens-update.story.md#237-387). Это нарушает правило команды запускать фронтенд через Docker и не даёт QA воспроизводимого процесса внутри контейнера.
2. **Accent-токены перечислены неодинаково в AC и Task 2.** Acceptance Criteria требуют обновить 7 значений (success/warning/danger/promo + их фоновые пары) (@docs/stories/epic-14/14.1.design-system-tokens-update.story.md#87-115), но Task 2 по-прежнему говорит лишь про «4 значения» (@docs/stories/epic-14/14.1.design-system-tokens-update.story.md#303-317). Это создаёт риск, что фоновые токены не попадут в scope и останутся старыми.
3. **Компонентные значения завязаны на внешний документ без конкретных критериев в самой истории.** AC3 ссылается на sprint-change-proposal для Checkbox/Toggle/Tabs/Toast (@docs/stories/epic-14/14.1.design-system-tokens-update.story.md#126-175) и Tasks повторяют эту ссылку (@docs/stories/epic-14/14.1.design-system-tokens-update.story.md#312-315). QA не может подтвердить миграцию без сравнения с отдельным файлом, что усложняет трассировку требований.
4. **Проверка tailwind.config.ts оформлена как «опциональная» заметка, а не критерий готовности.** Раздел Tailwind Config говорит, что конфиг нужно очистить от HEX, если они найдены, но это не отражено ни в AC, ни в DoD (@docs/stories/epic-14/14.1.design-system-tokens-update.story.md#530-564). Story может считаться завершённой, даже если конфиг продолжит переопределять старые цвета.

#### Recommendations

- Добавить инструкцию или ссылку на готовый workflow запуска `npm run build/dev` и JSON-валидации внутри Docker-контейнера фронтенда, чтобы проверки выполнялись единообразно.
- Синхронизировать описание accent-токенов: перечислить полный список (включая \*\_bg) и обновить Task 2, чтобы исключить частичную миграцию.
- Зафиксировать минимальный набор ожидаемых значений для Checkbox/Toggle/Tabs/Toast прямо в истории (таблица или вложение), чтобы QA мог сверить изменения без внешних ссылок.
- Повысить требование по tailwind.config.ts до Acceptance Criteria или DoD: добавить пункт, что конфиг не должен содержать хардкод HEX-палитры после миграции.
